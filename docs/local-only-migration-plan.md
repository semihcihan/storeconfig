# Local-Only Migration Plan

## Goal

Make StoreConfig usable as a local open-source tool without the hosted StoreConfig backend, AWS Serverless stack, Paddle billing, Mailgun emails, DynamoDB, S3, hosted accounts, or remote job polling.

The public version should let a user install the CLI, configure their own App Store Connect credentials locally, fetch their current App Store state, edit JSON, validate it, compare prices, and apply changes directly from their machine.

## Current Shape

The local-only migration now keeps these packages:

- `cli`: user-facing command line tool.
- `api`: core App Store Connect engine and business logic.
- `shared`: schemas, validation, models, helpers, logger.
- `app-store-connect-api-types`: generated/typed App Store Connect client definitions.
- `web`: local-only documentation/marketing site for the free open-source CLI.

The important local migration insight is that the App Store Connect logic mostly lives in `api`. The removed `serverless` package mainly wrapped that logic with hosted-product concerns: auth, persistence, async jobs, user records, subscriptions, email, and AWS infrastructure.

## Target Shape

For the local-only open-source version, keep:

- `cli`
- `api`
- `shared`
- `app-store-connect-api-types`
- `web`

Remove:

- `serverless`
- `admin-web`
- Paddle/payment flows
- Mailgun/API-key request flows
- hosted user/account concepts
- remote job/status polling

`web` should remain, but it must become a local-only documentation/marketing site. It should not include signups, beta/waitlist flows, hosted subscription management, secret-key request pages, or hosted account messaging.

## Main Architectural Changes

### 1. Local Apple Credentials

Current behavior:

- `storeconfig apple` sends Apple credentials to the hosted backend.
- The backend stores them and injects them into process env before calling App Store Connect.

Target behavior:

- `storeconfig apple --key-path <path>` stores the Apple key ID, issuer ID, and private key locally.
- The CLI/local engine loads those credentials directly when calling App Store Connect for user-specific app data and mutations.
- The local product has exactly one active Apple credential set. It does not support backup App Store Connect credentials, fallback `FALLBACK_ASC_*` environment variables, or server-side cron credentials for user workflows. Price-point metadata is the exception: it is refreshed from public static snapshots generated by the repository's GitHub Actions workflow.

Storage options:

- MVP: local config file under the user's config directory.
- Better: OS keychain or a clearly documented encrypted/permissioned file.

### 2. Direct Local Engine Calls

Current behavior:

- CLI commands call the hosted backend through `cli/src/services/api-client.ts`.

Target behavior:

- CLI commands call local services directly from `api`.

Important route replacements:

- `/fetch-apps` -> local `fetchAppsList`
- `/fetch` -> local `fetchApp`
- `/diff` -> local diff service
- `/apply` -> local apply service
- `/compare-price` -> local compare service
- `/price-points` -> local/cache-backed price point service backed by public static snapshots
- `/auth`, `/info`, `/status`, `/request-api-key`, `/generate-api-key` -> remove or replace with local state commands

### 3. Synchronous Apply

Current behavior:

- `storeconfig apply` creates a backend job.
- The backend persists job payload/state and invokes worker Lambdas.
- The CLI polls `/status/{jobId}`.

Target behavior:

- `storeconfig apply` runs in-process.
- Progress is printed directly by the CLI.
- MVP does not need remote job records or polling.

Possible later improvement:

- Local resumable job history under a cache/state directory.

### 4. Local Cache Instead Of AWS Storage

Current behavior:

- DynamoDB stores users, Apple keys, API-key requests, jobs, payment events.
- S3 stores large job payloads, price points, currencies, and related cached data.

Target behavior:

- Apple credentials: local secure config.
- Price points: local cache files refreshed from public static snapshots.
- Currencies: bundled data with local cache files.
- Current job/latest action: not required for MVP, optional local history later.
- User/subscription/payment/API-key request records: delete from the local product.

### 5. Remove Hosted Product Packages And Flows

The local-only product should delete the hosted product surface instead of carrying it as inactive code.

Remove:

- `serverless`
- `admin-web`
- Paddle webhook handling, subscription records, customer records, and payment event processing
- Mailgun clients, mailing list scripts, announcement scripts, and API-key request emails
- Hosted user model, API keys, authorizer context, `/info` user response, and account activation flows
- Remote jobs, job status polling, SQS/Lambda worker flow, DynamoDB job records, and S3 job payload storage

This also means the CLI should stop requiring `storeconfig configure` with a StoreConfig secret key. Local Apple credentials become the only required credential.

### 6. Web Site Repositioning

`web` stays, but its job changes.

Remove from `web`:

- waitlist/signup CTAs
- beta access messaging
- Paddle customer portal links
- secret-key request and secret-key reveal pages
- hosted pricing/subscription copy
- account-oriented FAQ/docs content

Keep or update in `web`:

- install instructions
- local Apple credential setup
- fetch/validate/apply workflow
- JSON schema docs
- examples
- MCP/AI usage docs
- public GitHub links

### 7. Package And Publishing Cleanup

Recommended public package shape:

- `storeconfig` CLI package includes the CLI and the local engine it needs.
- `serverless` is deleted.
- `admin-web` is deleted.
- Public docs explain local credential setup and direct App Store Connect API usage.

## Suggested Implementation Milestones

These milestones are not publish points. The project should not be published between them. They are ordered by testability and implementation dependency: each milestone should leave the codebase easier to run or verify locally than the previous one.

### Milestone 1: Local Credentials And Engine Boundary

- Add a local credential store abstraction.
- Change `storeconfig apple` to write local credentials.
- Add a local runtime adapter that prepares App Store Connect auth context for `api`.
- Keep command behavior mostly unchanged.
- Test local credential save/load without touching network services.

Status: complete. `storeconfig apple` now stores credentials locally, and the local auth adapter prepares `ASC_PRIVATE_KEY`, `ASC_KEY_ID`, and `ASC_ISSUER_ID` for the existing `api` auth context. Other CLI commands still use hosted backend calls until later milestones.

Expected difficulty: medium.

### Milestone 2: Replace Read-Only Backend Calls

- Replace `fetch-apps`.
- Replace `fetch`.
- Replace `compare-price`.
- Replace `price-points` with local cache-backed logic.
- Test commands that do not mutate App Store Connect state first.

Status: complete. `fetch`, `compare-price`, and the read-only pricing lookup used by `set-price` now call the local `api` engine directly through the CLI `storeconfig-engine` service. Apple credentials are loaded locally before user-specific App Store Connect calls. Price points are cached under `~/.storeconfig/cache/price-points` and refreshed from shared static snapshots generated weekly by GitHub Actions, currencies are loaded from bundled data with a local cache under `~/.storeconfig/cache/currencies.json`, and these caches expire after 7 days to match the old weekly serverless refresh cadence. These commands no longer require the hosted StoreConfig secret key. Hosted account commands and legacy hosted service files were removed in Milestone 4.

Expected difficulty: medium-low.

### Milestone 3: Replace Diff And Apply

- Replace `/diff` with direct local diff service calls.
- Replace `/apply` job creation with synchronous local apply.
- Remove CLI job polling for local mode.
- Preserve progress/error reporting.
- Test diff generation before enabling real apply execution.

Status: complete. `storeconfig apply` now generates the diff through the local `api` engine, shows the plan, prompts for confirmation, and applies the plan synchronously from the CLI process. The command no longer creates hosted jobs, polls remote job status, or requires a StoreConfig secret key. Apply progress is reported from the local action callback, and job info messages produced by the `api` layer are printed directly by the CLI. The old hosted job service and polling service were removed in Milestone 4.

Expected difficulty: medium-high.

### Milestone 4: Delete Hosted Product Surface

- Remove API key configuration requirements.
- Remove `user` command or redefine it as local credential/status info.
- Delete Paddle/Mailgun/API-key request flows.
- Delete `serverless`.
- Delete `admin-web`.
- Remove hosted-product workspaces from the root workspace list.
- Test workspace build/install after package removal.

Status: complete. The CLI no longer registers `configure` or `user`, no commands require a StoreConfig Secret Key, and the hosted API client, user, key, info, job, polling, and secret-key prompt services were removed. The MCP server no longer exposes hosted user/account tooling. `serverless` and `admin-web` were deleted, root workspaces were reduced to the local packages, and package metadata was updated for the local-only workspace.

Expected difficulty: medium.

### Milestone 5: Rework Web For Local-Only StoreConfig

- Remove signup/waitlist/subscription pages and CTAs.
- Remove secret-key request pages.
- Rewrite docs and marketing copy around local installation and direct App Store Connect usage.
- Keep schema docs, command docs, examples, blog content that still applies, and public GitHub links.
- Test web build and scan visible copy for hosted-product references.

Status: complete. The web app now presents StoreConfig as a free, open-source, local-only tool. Signup, waitlist, Firebase analytics, Paddle/payment config, StoreConfig secret-key pages, and hosted-account copy were removed. The homepage, docs, FAQ, about, legal pages, structured data, sitemap, environment example, and retained blog content now point users toward local installation, local Apple credential setup, direct App Store Connect usage, and the public GitHub repository.

Expected difficulty: medium-low.

### Milestone 6: Final Validation And Documentation

- Update README and CLI docs.
- Add tests around local credential loading and direct command execution.
- Run one final current-tree and history secret scan before publishing.
- Run full local validation once all hosted dependencies are removed.

Status: complete. Root README, CLI README, and Cursor project notes now describe the local-only open-source workflow. CLI command registration is centralized in a local command list, and tests cover both the local command surface and Apple credential loading before direct app-list/app-data execution. The stale root TypeScript project reference to deleted `serverless` was removed. Final validation passed with `npm run build`, `npm run test --workspaces --if-present -- --watchman=false`, and focused CLI command/engine tests. Current-tree and Git history secret scans found no live-looking secret values; remaining current-tree hits are empty env examples, test placeholder private-key fixtures, and migration notes, while history hits are old placeholder values such as `Content-of-the-dot-p8` and `your-storeconfig-api-key`.

Expected difficulty: medium.

## Effort Estimate

MVP local CLI:

- Around 3-5 focused development days.
- Includes local credentials, read commands, compare-price, and synchronous apply.

Good public release:

- Around 1-2 weeks.
- Includes tests, docs, package cleanup, cache handling, and removing hosted-product assumptions.

Highly polished local tool:

- Around 2-4 weeks.
- Includes keychain integration, resumable local jobs, richer progress reporting, robust cache invalidation, and migration docs.

## Main Risks

- `apply` currently depends on backend job orchestration and status polling. This is the largest migration point.
- Price point cache behavior no longer depends on S3 or user-owned representative resources. Local caches are refreshed lazily from GitHub Pages snapshots after a 7-day TTL, while the snapshots themselves are regenerated by a weekly GitHub Actions cron.
- Credential handling needs to be safe enough for an open-source tool.
- Some `api` services may assume process env auth context. The local adapter should make this explicit rather than scattering env writes through CLI code.

## Recommended Direction

Build the local version around the existing `api` package instead of reimplementing App Store Connect behavior in the CLI.

The migration should remove hosted-product concepts from the default user flow, not emulate the serverless backend locally. The simplest good product is a single local CLI that reads local Apple credentials, calls App Store Connect directly, and uses local cache/state only where it materially improves the workflow.
